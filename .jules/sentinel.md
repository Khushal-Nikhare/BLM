## 2024-05-18 - [CRITICAL] Fix hardcoded JWT secret fallback
**Vulnerability:** A hardcoded secret (`'default_secret'`) was used as a fallback for the `JWT_SECRET` environment variable in both `backend/routes/auth.js` and `backend/middleware/auth.js`. This allows an attacker to easily forge JWTs if the environment variable is not properly set or accidentally omitted in production.
**Learning:** In ES Modules, top-level code (like variable declarations using `process.env`) is evaluated when the module is imported. Since `auth.js` and `auth` middleware are imported in `index.js` *before* `dotenv.config()` is called, they might fetch `undefined` for `process.env.JWT_SECRET` locally, making the fallback kick in even if it is defined in `.env`. The fix is to dynamically evaluate the environment variable inside the function execution block (e.g. inside `router.post('/login')` and `verifyToken`) and to fail securely by returning a `500` error if it is missing, rather than falling back to an insecure default.
**Prevention:**
1. Never use hardcoded fallback values for secrets. Fail securely with a loud error.
2. In Node.js applications using ES Modules and `dotenv`, fetch critical environment variables dynamically inside functions when they are needed, or ensure `dotenv/config` is imported as the very first import before any other modules that depend on environment variables.
